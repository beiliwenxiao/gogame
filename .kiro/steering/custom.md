## 代码探索方法论

### 搜索陌生功能时的正确姿势
- 不要用视觉效果名称搜索（如 `renderAttackRange`），要从**数据/状态**角度切入（如 `selectedTarget`、`dead`）
- 对不知道关键词的情况，优先用 `readCode` 看类的方法签名列表，再按方法名逐个深入，比盲目 `grepSearch` 效率高得多

### 状态限制的完整性原则
给某个状态（如灵魂状态、眩晕等）加限制时，需要覆盖所有入口层：
1. 执行层（实际发送消息/造成效果的函数）
2. 交互层（选中目标、点击按钮等 UI 入口）
3. 状态清理（进入该状态时清除残留的交互状态，如 `selectedTarget`）

漏掉任何一层都会有残留的视觉或交互 bug。

## Go 后端调试技巧

- `go build . 2>&1` 在 Windows bash 里可能 exit code -1 但无输出（shell 重定向问题）
- 检查 Go 文件语法优先用 `getDiagnostics` 工具，比跑 shell 命令更可靠


### Bug 排查：后端数据正确但前端表现异常时
- 确认后端数据无误后，立刻看 `BaseGameScene.update` 的调用链，逐个检查父类系统（CombatSystem、MeditationSystem 等）是否有修改相关状态的逻辑
- 从 `update` 调用链自顶向下排查，比盲目 grep 关键词高效得多
- 特别注意单机引擎自带的逻辑（如自动复活、自动回血）在多人联网场景下可能产生冲突


## 联网/单机行为分离模式

当单机引擎的某个系统（如 CombatSystem）有自动逻辑（自动复活、自动回血等）与联网场景冲突时：
1. 在系统上设 `_arenaMode` 布尔标志
2. 联网场景 `enter` 时设 `true`，`exit` 时清 `false`
3. 系统内部检测标志，跳过单机专属逻辑

比 `_safeZoneDisabled`（每帧设/清）更适合场景级别的持久开关。

## 2.5D 椭圆判定一致性

渲染用椭圆时，碰撞判定也必须用椭圆公式 `(dx/rx)² + (dy/ry)² <= 1`，不能用圆形近似。前后端必须同步修改判定逻辑。


## 对话续接时的代码验证原则

上下文续接时，上一轮"已完成"的修改可能实际上只有调用点，没有实现体。续接后必须用 `grepSearch` 验证关键方法是否真实存在，不能只看调用处就认为已完成。


## editCode replace_node 对缩进敏感

- `editCode replace_node` 替换方法体时，如果 replacement 的缩进与文件中实际缩进不匹配，会 "Write verification failed"
- 常见场景：类方法在文件中是 8 空格缩进（双层），replacement 写成 4 空格
- 遇到失败时改用 `strReplace` 精确匹配，或先用 `readFile` 确认实际缩进再写 replacement


## 战斗 UI 效果的方案确认原则

实现前先确认"触发条件"和"显示对象"两个维度，避免反复返工：
- 触发条件：从玩家实际操作场景出发（鼠标在世界区域还是 UI 上？是持续显示还是事件触发？）
- 显示对象：哪些技能/职业需要显示？跟随谁？圆心在哪？
- 战斗类持续显示效果不能用 hover 触发（玩家鼠标在世界区域打怪，不会悬停 UI）
- 近战范围指示器跟随玩家有意义，远程技能以目标点为中心跟随玩家没意义


## 玩家 vs NPC 状态效果的驱动模式差异

- NPC 状态效果（恐惧逃跑、昏迷等）由后端 `npcAITick` 驱动（服务端权威移动）
- 玩家状态效果由前端 `ArenaScene.update` 驱动（客户端控制移动 + `sendMovement` 同步位置到后端）
- 同一个效果（如"恐惧"），前后端实现路径完全不同：NPC 改后端 AI 循环，玩家改前端 update 循环
- 新增玩家状态效果时，需要在 `ArenaScene.update` 中加入状态检查 + 自动行为 + 禁用攻击/技能入口

## 多同步通道下的防护失效陷阱

"前端忽略了某条消息但效果仍然生效"时，要检查是否有其他同步通道（如 `doStateSync` 增量同步）绕过了前端的消息级防护。WebSocket 消息级别的拦截 ≠ 状态级别的拦截。后端是权威数据源，只要后端状态变了，任何周期性同步机制都会把变化推到前端。修复必须从后端源头阻断，而非在前端消息处理层做过滤。


## 联网模式下单机引擎的"隐式伤害源"排查

当"后端实体还活着但前端认为死了"时，除了检查消息处理（`onDamage`/`onNPCDied`），还要排查单机引擎的每帧逻辑是否在联网模式下仍对实体造成本地伤害：
1. `CombatSystem.checkDeath` — 每帧检测 `stats.hp <= 0` 触发 `handleDeath`
2. `MeleeAttackSystem.updateSectorSlashEffects` — 刀光/箭矢碰撞检测调用 `applyDamage`
3. 其他通过 `super.update()` 调用链触发的单机战斗逻辑

通用规律：联网模式下，所有"本地造成伤害"和"本地判定死亡"的逻辑都必须被 `_arenaMode` 守卫。视觉特效和碰撞伤害耦合在同一对象中时，联网模式需要在碰撞层拦截而非禁用整个特效。


## 前后端基本逻辑流程
1. 玩家操作（如点击技能图标）
2. 技能图标触发技能（如技能图标的 `onClick`）
3. 当前玩家技能直接触发视觉效果，但不执行伤害。
4. 技能图标调用后端 API，发送技能消息给后端（如 `sendSkill`）
5. 后端收到消息后，计算技能效果，并广播给所有玩家。
当前玩家不需要再次播放技能，只需要同步所有玩家数据，技能状态更新。
6. 技能效果被所有玩家的前端接收，并触发前端动作。
数据更新，技能状态更新。
7.如果用到goroutine锁，则一次性在锁内完成所有计算和状态修改（包括恐惧方向、FearUntil等等），然后统一 Unlock，锁外做所有广播。
